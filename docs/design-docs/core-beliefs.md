# Core Beliefs — Interview Connect 설계 철학

`domain-designer`, `product-analyst`가 설계 결정 시 이 원칙을 따른다.

---

## 제품 철학

### 1. 연결 > 정보
정보 나열이 아니라 사람을 연결한다.
후기는 연결의 시작점이고, Q&A는 연결의 매개체다.
기술적 구현도 이 방향을 지원해야 한다.

### 2. 신뢰는 인증에서 온다
아무나 쓴 후기는 신뢰하기 어렵다.
VERIFIED 회원만 후기를 작성할 수 있는 이유가 여기에 있다.
Phase 1에서 VERIFIED 등급이 비어 있어도, 구조는 미리 만들어둔다.

### 3. 공개로 유입, 블러로 전환
비로그인에게 최대한 많은 콘텐츠를 공개하여 검색 유입을 만든다.
그러나 Q&A 답변만 블러 처리하여 가입 동기를 만든다.
이 균형을 깨는 구현(후기 공개 축소, 블러 범위 확대)은 전략에 반한다.

### 4. 작성자 보호
인증된 경험자가 계속 참여하려면 보호받는다고 느껴야 한다.
1:1 쪽지 수락제, 익명 질문 옵션, 요청 제한이 이를 위한 장치다.

## 기술 설계 원칙

### 도메인 중심 설계 (DDD)
- 비즈니스 로직은 엔티티와 도메인 서비스에 위치한다
- Controller는 라우팅과 DTO 변환만 담당한다
- Service는 트랜잭션 경계와 유스케이스 조합을 담당한다

### 단순함 우선
- 복잡한 추상화보다 명확한 구현을 선호한다
- Phase 1에서 Phase 3 수준의 확장성을 구현하지 않는다
- 씨드 데이터조차 가장 단순한 방법으로 구현한다 (DataInitializer)

### 테스트가 명세다
- 구현 전 테스트가 "어떻게 동작해야 하는가"를 정의한다
- 테스트의 @DisplayName은 비즈니스 요구사항을 표현한다
- 테스트가 통과하면 구현이 올바른 것이다

### 보안은 기본값
- 새 API를 추가할 때 인증 요구 여부를 먼저 결정한다
- permitAll이 아닌 한 기본적으로 인증을 요구한다
- 비밀번호는 항상 해싱, 토큰은 항상 검증
