=== 엔티티 작성 규칙 ===

[필수 어노테이션]
@Entity
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
BaseTimeEntity 상속 (createdAt, updatedAt 자동)

[정적 팩터리 메서드 패턴 — Setter 대신 사용]
@Builder를 private 생성자에 붙인다.
of() 또는 from() 정적 메서드로 생성한다.

예시:
  @Builder
  private Member(Long id, String email, String password, String nickname, MemberRole role) {
      this.id = id;
      this.email = email;
      this.password = password;
      this.nickname = nickname;
      this.role = role;
  }

  public static Member createGeneral(String email, String password, String nickname) {
      return Member.builder()
          .email(email)
          .password(password)
          .nickname(nickname)
          .role(MemberRole.GENERAL)
          .build();
  }

[값 변경 — change 메서드]
Setter 사용 금지. 상태 변경은 명시적 change 메서드를 만든다.
  public void changeRole(MemberRole newRole) {
      this.role = newRole;
  }

[Enum 체크 — Enum에게 위임]
if (role == MemberRole.VERIFIED) → 금지
role.isVerified() 처럼 Enum 내부에 메서드를 만들어 사용한다.

=== 서비스 작성 규칙 ===

@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)  // 기본값
// 쓰기 메서드에만 @Transactional (readOnly 없이)

public class MemberService {
    private final MemberRepository memberRepository;

    // 조회 — @Transactional(readOnly = true) 상속
    public Member findById(Long id) {
        return memberRepository.findById(id)
            .orElseThrow(() -> BusinessException.from(ErrorCode.MEMBER_NOT_FOUND));
    }

    // 쓰기 — @Transactional 추가
    @Transactional
    public void changeRole(Long memberId, MemberRole role) {
        Member member = findById(memberId);
        member.changeRole(role);
    }
}

=== 컨트롤러 작성 규칙 ===

@RestController
@RequestMapping("/api/v1/{도메인}")
@RequiredArgsConstructor

응답은 항상 ApiResponse<T> 래퍼:
  return ApiResponse.ok(responseDto);
  return ApiResponse.error(ErrorCode.XXX, message);

=== 조건문 작성 규칙 ===

[얼리 리턴 — if-else 대신]
나쁜 예:
  if (condition) {
      // do something
  } else {
      // do something else
  }

좋은 예:
  if (isInvalid(condition)) {
      throw BusinessException.from(ErrorCode.INVALID_INPUT);
  }
  // 정상 로직

[부정 조건 — 라이브러리 활용]
!something.equals(x) → Objects.nonNull(x) 패턴 활용
null 체크: Objects.isNull(x), Objects.nonNull(x)
비어있는 Optional: optional.isEmpty()

[중첩 if → 함수 추출]
중첩 depth 2 이상이면 별도 메서드로 추출한다.

=== 숫자 타입 ===

금액/비율 등 정밀도가 필요한 값: BigDecimal 사용
Double, Float 사용 금지 (부동소수점 오차)

=== final 키워드 ===

변하면 안 되는 모든 값에 final:
  private final MemberRepository memberRepository;
  final String email = request.email();

=== 함수 순서 ===

파일 내 메서드는 호출 순서대로 위에서 아래로 배치한다.
호출자 메서드가 피호출자 메서드보다 위에 있어야 한다.

=== 주석과 로그 ===

주석: 한글로 작성 (비즈니스 로직 설명)
로그: 영어로 작성 (log.info("User registered: {}", email))
불필요한 주석 금지, 코드로 설명 가능한 내용은 주석 생략

=== import 관리 ===

사용하지 않는 import 제거 (IDE 자동 정리 or 빌드 시 확인)
와일드카드 import 금지 (java.util.* 금지)

=== DTO 스타일 ===

record 사용 권장:
  public record SignupRequest(
      @NotBlank String email,
      @Size(min = 8) String password,
      @NotBlank @Size(max = 30) String nickname
  ) {}

Request/Response 분리 — 하나의 DTO를 양방향으로 사용하지 않는다.

=== Repository ===

도메인 계층에 인터페이스만:
  public interface MemberRepository {
      Member save(Member member);
      Optional<Member> findById(Long id);
      Optional<Member> findByEmail(String email);
      boolean existsByEmail(String email);
      boolean existsByNickname(String nickname);
  }

JPA 구현은 ic-infra 또는 ic-domain의 JpaRepository 확장으로 별도 분리.

=== 예외 처리 ===

BusinessException.from(ErrorCode) — 비즈니스 예외
BusinessException.of(ErrorCode, message) — 커스텀 메시지

GlobalExceptionHandler가 @RestControllerAdvice로 ApiResponse<Error> 변환.
try-catch 남용 금지. 예외는 위로 전파하고 핸들러에서 처리.

=== 테스트에서 사용하지 않는 것 ===

@SpringBootTest 기본 사용 금지 (느림)
@MockBean 남용 금지
Fake 구현체를 우선 사용한다.
